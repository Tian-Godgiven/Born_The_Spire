# 效果开发指南

效果（Effect）是游戏逻辑的最小执行单元。本文档说明如何设计效果，以及在实现新内容时如何在"复用"与"新建"之间做出正确选择。

---

## 效果的三个层次

### 原子效果

单一职责，只做一件事，通过 params 参数化。这类效果可以被任意内容复用。

```
damage              造成伤害
heal                回复生命
applyState          施加状态
removeState         移除状态
changeStateStack    修改状态层数
gainArmor           获得护甲
drawCard            抽牌
discard             弃牌
addStatusBase       修改基础属性（加减）
setBaseStatus       直接设置基础属性
addCurrent          修改当前值
loseHealth          失去生命（绕过伤害事件）
kill / revive       击杀/复活
gainCard / gainOrgan / gainRelic / gainPotion
```

完整列表见 `src/static/list/system/effectMap.ts`。

### 复合效果

封装一类常见模式，通过 params 驱动行为。不针对任何特定内容，可跨器官/卡牌/遗物复用。

```
accumulateAndTrigger    积累数值，达到阈值时触发效果
countAndTrigger         计数器，达到阈值时触发效果
gainArmorPerAlly        按友方数量获得护甲
repeatEffects           重复执行效果 N 次
stuffCard               向目标牌堆塞入卡牌（玩家/敌人通用）
addCardToHand           向手牌添加卡牌
addTemporaryEffect      给卡牌添加临时效果
giveTemporaryEffectToRandomCards  给随机卡牌添加临时效果
disableRandomOrgans     随机禁用器官
```

### 专属效果

逻辑仅适用于某一个具体内容，命名带 `organ_` 或 `card_` 前缀，不应被其他内容引用。

```
organ_heatTick          炉心热量累积
organ_heatAccumulate    热量积累触发
card_heatBlast          热爆：消耗热量造成伤害
card_commandStrike      指挥打击
organ_lifeSteal         腐食再生
```

## 决策流程

实现一个新效果前，按以下顺序检查：

第一步：能否用现有原子效果 + params 直接实现？

```typescript
// 例：造成 5 点伤害 → 直接用 damage
{ key: "damage", params: { value: 5 } }

// 例：施加 2 层中毒 → 直接用 applyState
{ key: "applyState", params: { stateKey: "poison", stacks: 2 } }
```

第二步：能否用现有复合效果实现？

```typescript
// 例：每受到 8 点伤害就回复 2 点 → accumulateAndTrigger
{
    key: "accumulateAndTrigger",
    params: {
        pointKey: "healAccum",
        on: { when: "after", how: "take", key: "damage" },
        gain: "$triggerEffect.params(value)",
        threshold: 8,
        consume: 8,
        effects: [{ key: "heal", params: { value: 2 } }]
    }
}
```

第三步：同样的模式是否在其他地方出现过（或预计会出现）？

如果同一逻辑出现了两次以上，考虑将其提炼为新的复合效果，放在 `src/core/effects/` 下的合适目录，并注册到 effectMap。

第四步：以上都不行，写专属效果。

以 `organ_` 或 `card_` 前缀命名，放在 `organSpecificEffects.ts` 或 `cardSpecificEffects.ts`，不要被其他内容引用。

## 反例分析

以下是 Boss 开发中实际出现的问题，作为参考。

### organ_mycelialSpread（菌网扩散）

效果：给随机一个对手施加 N 层中毒。

写成了专属效果，但实际上本质是 `applyState` 的变体——唯一区别是目标是"随机对手"而非固定目标。

```typescript
// 写的代码：专属效果，内部自己选取随机对手再 doEvent
export const organ_mycelialSpread: EffectFunc = (event, effect) => {
    const opponents = isEnemy(source) ? battle.getAlivePlayers() : battle.getAliveEnemies()
    const random = opponents[Math.floor(rng.nextFloatRange(0, opponents.length))]
    doEvent({ key: "applyState", target: random, ... })
}
```

正确做法：当效果仅因为目标选取逻辑而无法复用时，应扩展目标解析系统（targetType），而不是写专属效果。

```
// 理想形式（需扩展 targetType 支持 randomOpponent）
{ key: "applyState", params: { stateKey: "poison", stacks: 2, targetType: "randomOpponent" } }
```

### organ_lifeDrain（寄生菌根）—— 已修复

效果：对手每累计受到 8 点伤害，持有者回复 2 点生命。

初版写了专属效果 `organ_lifeDrain`，把计数器存在 StateModifier（状态系统）上，重复实现了 `accumulateAndTrigger` 的核心逻辑。

修复后：给器官 status 加 `lifeDrainAccum: 0`，在 `possess.effects` 里直接用 `accumulateAndTrigger`，同时给它加了 `triggerTarget: "allOpponents"` 参数支持，使其能将触发器挂到对手身上。删除了 `organ_lifeDrain` 和 `stateList` 里的 `lifeDrainAccum` 状态。

### organ_corruptionArmor（腐化铠甲）

效果：回合结束时，每有 1 个对手具备 debuff，获得 N 点护甲。

目前作为专属效果存在是合理的——"统计满足条件的对手数量并据此获得护甲"暂时只出现了一次。但如果后续有第二个器官需要类似逻辑（如：每有 1 个对手处于燃烧状态，获得 2 点护甲），应将其提炼为通用的 `gainArmorPerOpponentCondition` 复合效果。

## 专属效果的合理场景

以下情况写专属效果是合理的：

效果依赖某个独特系统（热量、腐蚀、充能等）
  这类系统的状态机复杂，参数化困难，且不打算在其他地方出现相同机制

效果有复杂的多步状态机
  例如 `organ_heatTick`：检查热量 → 累积热量 → 超过阈值 → 冷却

效果的逻辑与某张卡牌/器官高度耦合
  例如 `card_commandStrike`：逻辑依赖"指挥"标签的具体语义

---

## 命名规范

原子效果：动词 + 名词，如 `gainArmor`、`applyState`、`drawCard`
复合效果：动词短语，如 `accumulateAndTrigger`、`countAndTrigger`、`gainArmorPerAlly`
专属效果：`organ_` 或 `card_` + 驼峰名称，如 `organ_heatTick`、`card_commandStrike`

专属效果的 label（在 effectMap 中）应描述其所属内容：

```typescript
{ label: "腐化铠甲：对手有debuff时获得护甲", key: "organ_corruptionArmor", effect: organ_corruptionArmor }
```
