# 循环依赖解决任务清单

## 优先级说明

P0 - 核心循环，必须立即解决
P1 - 重要循环，影响架构清晰度
P2 - 次要循环，可以后续优化

---

## P0 - 核心系统循环（3个）

### 任务 1: ActionEvent ↔ Effect 循环
**状态**: 待处理
**难度**: 中等
**影响范围**: 核心事件系统

**循环路径**:
```
ActionEvent.ts → Effect.ts → ActionEvent.ts
```

**问题描述**:
  ActionEvent 需要创建 Effect 对象
  Effect 需要引用 ActionEvent 类型（用于级联事件）

**解决方案**:
  方案A: 将 Effect 的 ActionEvent 引用改为接口类型
  方案B: 使用工厂函数（通过 container）注入 ActionEvent 创建能力
  方案C: 将级联事件创建逻辑移到 ActionEvent 内部

**推荐**: 方案A + 方案B 组合

---

### 任务 2: Effect ↔ EffectFunc 循环
**状态**: 待处理
**难度**: 中等
**影响范围**: 效果执行系统

**循环路径**:
```
Effect.ts → EffectFunc.ts → Effect.ts
```

**问题描述**:
  Effect 需要调用 EffectFunc 执行效果
  EffectFunc 需要创建新的 Effect 对象（级联效果）

**解决方案**:
  方案A: EffectFunc 返回 EffectUnit 数据，由调用方创建 Effect
  方案B: 通过 container 注入 Effect 工厂函数
  方案C: 将 EffectFunc 改为纯函数，不创建 Effect

**推荐**: 方案A（符合数据驱动原则）

---

### 任务 3: Entity → ActionEvent → EffectFunc → typeGuards → State → Target → Entity 长循环
**状态**: 待处理
**难度**: 高
**影响范围**: 整个核心架构

**循环路径**:
```
Entity.ts → ActionEvent.ts → Effect.ts → EffectFunc.ts
→ typeGuards.ts → State.ts → Target.ts → Entity.ts
```

**问题描述**:
  这是最长的循环链，涉及所有核心类
  typeGuards 作为工具函数，却需要导入所有类型

**解决方案**:
  方案A: 拆分 typeGuards.ts
    typeGuards/base.ts - 只检查基础类型
    typeGuards/entity.ts - 检查 Entity 相关
    typeGuards/item.ts - 检查 Item 相关
  方案B: 使用类型谓词（type predicates）+ 接口，避免导入具体类
  方案C: 将类型检查逻辑移到各个类的静态方法中

**推荐**: 方案A + 方案B 组合

---

## P1 - 数据配置循环（2个）

### 任务 4: EffectUnit → lazyLoader → effectMap 循环
**状态**: ✅ 已完成（运行时）
**解决方案**:
  使用 lazyLoader 延迟加载 effectMap，运行时无循环依赖问题
  静态分析工具仍会报告，但这不影响实际运行

**说明**:
  运行时：EffectUnit 模块加载时不加载 effectMap，只有调用 getLazyModule('effectMap') 时才加载
  此时所有模块已初始化完成，无循环问题

---

### 任务 5: Current → currentMap → energy → Player 循环
**状态**: ✅ 已完成
**解决方案**:
  1. CurrentMapData.ts 将所有 `import` 改为 `import type`
  2. energy.ts 使用 `import type { Player }` 保持类型准确性
  3. 类型导入不会形成运行时循环依赖

---

## P2 - UI 层循环（2个）

### 任务 6: Item → describe → Status → Entity 循环
**状态**: ✅ 已完成
**解决方案**:
  1. describe.ts 使用 `import type { Status }` 而不是 `import { Status }`
  2. 给 Status 添加 `statusType = 'status'` 类型标识
  3. 在 typeGuards.ts 中添加 `isStatus()` 函数
  4. describe.ts 使用 `isStatus()` 替代 `instanceof Status`

---

### 任务 7: Trigger 子类循环
**状态**: ✅ 已完成
**解决方案**: 将 Trigger.ts 中对 defaultTrigger.ts 和 importantTrigger.ts 的函数导入改为 `import type`，保留类型定义，函数实现通过其他方式组织

npx madge --circular --extensions ts --image circular.png src/core

## 注意事项

  保持向后兼容：不要破坏现有的 mod 接口
  优先使用接口和类型：减少具体类的导入
  利用现有工具：container、lazyLoader 已经存在
  遵循项目原则：数据驱动、mod 友好、非程序员可用
  每次只解决一个循环：避免大规模重构导致的问题
