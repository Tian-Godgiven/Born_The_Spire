# 触发器 Trigger

触发器是绑定在实体Entity上，对过程事件ActionEvent/效果Effect的响应机制。

## 核心概念

触发器系统采用事件驱动架构，包含两个核心概念：

1. **触发器 (Trigger)**：负责"何时通知" - 监听特定事件并发出通知
2. **反应 (Reaction)**：负责"如何响应" - 定义收到通知后执行的具体行为

设计理念：将"触发时机"与"响应行为"分离，使触发器仅负责事件监测，响应行为由事件来源对象定义。

## 触发器结构

触发器通过【触发时机when】×【触发机制how】×【触发关键词key】三轴分类：

```typescript
{
    before: {  // 事件执行前
        make: { key: TriggerUnit[] }   // 来源实体
        via: { key: TriggerUnit[] }    // 媒介实体
        take: { key: TriggerUnit[] }   // 目标实体
    }
    after: {   // 事件执行后
        make: { key: TriggerUnit[] }
        via: { key: TriggerUnit[] }
        take: { key: TriggerUnit[] }
    }
}
```

### when: 触发时机

- **before**: 事件效果执行前，可修改事件参数
- **after**: 事件效果执行后，用于响应和联动

### how: 实体在事件中的角色

- **make**: 事件的发起者（造成事件）
- **via**: 事件的参与者/媒介（使用卡牌、发动攻击等）
- **take**: 事件的承受者（受到伤害、被治疗等）

### key: 触发关键词

与ActionEvent的key对应，例如：
- `damage`: 造成伤害
- `heal`: 造成回复
- `drawCard`: 抽卡
- `turnStart`: 回合开始
- `turnEnd`: 回合结束
- `battleEnd`: 战斗结束

### level: 触发优先级

数值越大越先触发，默认为0。用于控制多个触发器的执行顺序。

## 触发器数据格式 TriggerMap

TriggerMap是声明式的数据格式，用于在对象定义时配置触发器：

```typescript
{
    trigger: [{
        when?: "before" | "after",  // 触发时机，默认 before
        how: "make" | "via" | "take",
        key: string,                // 触发关键词
        level?: number,             // 优先级
        action: string,             // 指定响应名称（必需）
        info?: string,              // 描述信息
        condition?: TriggerCondition, // 触发条件（可选）
        triggerTarget?: {           // 指定触发器挂载目标（可选）
            participantType: "entity"
            key: string
        }
        timing?: "immediate" | "battleStart"  // 挂载时机
        disableUntil?: "battleEnd"             // 触发后禁用
    }]
}
```

### condition: 触发条件

可选的前置条件，满足条件才执行响应：

```typescript
condition: {
    sourceStatus?: {
        key: string       // 检查来源的status key
        value: number     // 期望值
        op?: "eq" | "lte" | "gte" | "lt" | "gt"
    }
    ownerHealthPercent?: {
        value: number     // 持有者生命百分比
        op?: "eq" | "lte" | "gte" | "lt" | "gt"
    }
}
```

## 反应系统 Reaction

反应是action名称到事件配置数组的映射，定义收到通知后执行的行为：

```typescript
{
    reaction: {
        "actionName": [
            {
                key: string,                        // 新事件的key
                label?: string,                     // 标签
                info?: Record<string, any>,         // 事件信息
                targetType: "eventSource" |         // 事件目标类型
                            "eventMedium" |
                            "eventTarget" |
                            "triggerSource" |
                            "triggerOwner" |
                            "triggerEffect" |
                            "owner" |
                            "randomEnemy" |
                            Entity,
                mediumType?: "source" |             // 事件媒介类型（可选）
                             "target" |
                             "triggerEventMedium" |
                             "triggerEffect",
                effect: EffectUnit[]                // 效果单元数组
            }
        ]
    }
}
```

### targetType: 事件目标类型

- **eventSource**: 原始事件的source对象
- **eventMedium**: 原始事件的medium对象
- **eventTarget**: 原始事件的target对象
- **triggerSource**: 为实体绑定触发器的来源对象
- **triggerOwner / owner**: 持有该触发器的对象（通常为玩家）
- **triggerEffect**: 触发该触发器的效果对象
- **randomEnemy**: 当前战斗中随机一个存活的敌人
- **Entity**: 指定的实体对象

### mediumType: 事件媒介类型

- **source**: 使用触发器的source（item/relic/organ）
- **target**: 使用触发器的target（默认，即owner）
- **triggerEventMedium**: 使用原始事件的medium
- **triggerEffect**: 使用触发效果（如damage效果）

## 工作流程

### 完整调用链

```mermaid
graph LR
    A[事件发生] --> B[ActionEvent执行]
    B --> C{触发器通知}
    C -->|匹配when/how/key| D[source.reaction[action]]
    D --> E[executeReactionEvents]
    E --> F[根据targetType解析目标]
    E --> G[根据mediumType解析媒介]
    E --> H[执行effect效果]
```

### 示例流程

遗物"回血石"在战斗结束时回复玩家生命：

```typescript
// 1. 遗物定义触发器
{
    key: "original_relic_00001",
    interaction: {
        possess: {
            target: { key: "self" },
            effects: [],
            triggers: [{
                when: "after",       // 事件执行后
                how: "make",         // 当owner发起此事件
                key: "battleEnd",    // 监听战斗结束事件
                action: "healOwner"  // 调用owner的healOwner反应
            }]
        }
    }
}

// 2. 玩家定义反应
{
    reaction: {
        healOwner: [{
            key: "healEvent",
            targetType: "owner",    // 目标是玩家自己
            effect: [{
                key: "heal",
                params: { value: 5 } // 恢复5点生命
            }]
        }]
    }
}
```

### 执行步骤

1. 战斗结束时创建 `battleEnd` 事件
2. 事件执行 `after make` 触发器
3. 找到遗物的 `trigger[action]` → `"healOwner"`
4. 调用 `executeReactionEvents`：
   - 解析targetType → 得到owner（玩家）
   - 解析mediumType → 默认为owner
   - 执行effect → 创建并执行 `heal` 效果
5. 玩家生命值增加

## 触发器类型

### 默认触发器 Default Triggers

实体创建时自动获得的触发器，记录在 `_defaultTrigger` 属性中：

```typescript
_defaultTrigger: [
    {
        triggerWay: "after_via_useCard",
        id: string,
        remove: () => void,
        info: string
    }
]
```

使用场景：
- 卡牌使用后自动弃置
- 回合开始自动抽牌

### 关键触发器 Important Triggers

可能与现有触发器冲突的触发器，使用 `importantKey` 和 `onlyKey` 管理：

```typescript
{
    when: "after",
    how: "via",
    key: "useCard",
    importantKey: "afterUseCard",  // 分组key
    onlyKey: "discardOrExhaust",   // 唯一性标识
    action: "discardOrExhaust"
}
```

使用场景：
- 卡牌添加"消耗"词条时替换原有的"弃置"行为
- 通过 `onlyKey` 确保同一时间只有一个触发器生效

## 创建触发器的方式

### 方式一：数据驱动（推荐）

在对象定义中使用 `trigger` + `reaction`：

```typescript
const relic = {
    key: "my_relic",
    interaction: {
        possess: {
            target: { key: "owner" },
            triggers: [{
                when: "after",
                how: "take",
                key: "damage",
                action: "reduceDamage"
            }],
            reaction: {
                reduceDamage: [{
                    key: "reduceDamageEvent",
                    targetType: "owner",
                    effect: [{
                        key: "modifyDamage",
                        params: { value: -2 }
                    }]
                }]
            }
        }
    }
}
```

### 方式二：代码动态添加

```typescript
// 在 Chara/Item/Organ 初始化时
entity.trigger.appendTrigger({
    when: "before",
    how: "take",
    key: "damage",
    callback: (event, effect, level) => {
        event.effects[0].params.value -= 2
    }
})
```

### 方式三：使用 TriggerMap API

```typescript
// 为实体添加触发器
applyTriggerMap(entity, [{
    when: "after",
    how: "make",
    key: "turnStart",
    action: "recoverEnergy"
}])
```

## 受支持的对象类型

### Chara（角色/敌人）

支持 `trigger` 和 `reaction`：

```typescript
{
    key: "player_default",
    trigger: [...],
    reaction: {...}
}
```

### Item（卡牌/药水/遗物）

支持 `trigger` 和 `reaction`：

```typescript
{
    key: "card_attack",
    trigger: [{
        when: "after",
        how: "via",
        key: "useCard",
        action: "discardToDiscardPile"
    }],
    reaction: {...}
}
```

### Organ（器官）

支持 `trigger` 和 `reaction`：

```typescript
{
    key: "organ_attack_boost",
    trigger: [...],
    reaction: {
        gainStrength: [{
            targetType: "owner",
            effect: [{ key: "addStatus", ... }]
        }]
    }
}
```

## 卸载触发器

### 通过remove函数

```typescript
const { remove, id } = entity.trigger.appendTrigger({...})
remove()  // 卸载该触发器
```

### 通过importantKey批量移除

```typescript
const triggers = entity.trigger.getImportantTrigger("afterUseCard")
triggers.forEach(t => t.remove())
```

### 通过onlyKey交换触发器

```typescript
// 自动移除旧触发器并添加新触发器
card.trigger.swapOnlyTrigger({...})
```

## 注意事项

1. **触发器仅通知，不执行行为**
   - 触发器的 `action` 字段指定响应名称
   - 实际行为定义在 `reaction` 中
   - 不要直接在callback中修改游戏状态

2. **⚠️ 重要：物品触发器的挂载原则**
   - **挂载位置**：物品的触发器默认挂载到物品的拥有者（`this.owner`）身上
   - `this.owner` 的含义：
     - 遗物的拥有者 = 玩家（遗物在玩家身上）
     - 器官的拥有者 = 敌人（器官在敌人身上）
   - **为什么挂载到拥有者？**：因为要监听拥有者的事件
     - 玩家抽牌时，事件由玩家发出（how: make）
     - 玩家回合开始时，事件由玩家发出（how: make）
     - 所以监听这些事件的触发器必须挂载在玩家身上

   ```typescript
   // 遗物配置示例
   {
       interaction: {
           possess: {
               triggers: [{
                   when: "after",
                   how: "make",          // 玩家发起的事件
                   key: "drawCard",      // 监听玩家的抽牌事件
                   action: "myAction"    // 在遗物的 reaction 中查找
                   // 触发器挂载到玩家身上（遗物的拥有者）
               }]
           }
       },
       reaction: {
           myAction: [...]  // 遗物的反应定义
       }
   }
   ```

   **设计优势**：
   - 事件监听在拥有者身上，能够捕获所有相关事件
   - 条件判断（condition）检查物品自身状态，保持逻辑封装
   - 响应动作（action）在物品的 reaction 中，模块化清晰
   - 物品被移除时，所有触发器自动清理

3. **targetType解析**
   - triggerOwner = 持有触发器的对象（通常是player）
   - triggerSource = 为对象绑定触发器的来源（通常是relic/organ）
   - eventTarget = 原始事件的target

4. **mediumType默认值**
   - 未指定时默认使用 "target"（即 owner）
   - 根据实际需求选择合适的medium

5. **condition检查顺序**
   - sourceStatus 检查在最前（检查物品自身状态）
   - ownerHealthPercent 检查在后（检查拥有者生命）
   - 任意条件不满足则不执行响应

6. **性能考虑**
   - 触发器回调应保持轻量
   - 复杂逻辑可分发到新的事件
