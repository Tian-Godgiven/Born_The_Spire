# 触发器系统重构计划

## 目标

将触发器系统从"触发器定义行为"改为"触发器只通知，行为由监听者自己定义"。

核心变化：
  当前：trigger 包含 event 数组，直接执行
  新设计：trigger 只负责监听，通知 source（遗物/器官），source 自己定义 reaction

## 设计确认

1. `trigger.source` = 遗物/器官（触发器提供者）
2. `trigger.owner` = 玩家（触发器挂载者）
3. trigger 仍然挂载到玩家身上
4. 执行时通知 `source`，由 source 的 `reaction` 定义响应
5. `reaction` 在遗物数据的顶层，和 `interaction` 平级
6. 保留 `targetType` 枚举，复用现有解析逻辑

## 新的数据结构

### 遗物定义示例

```typescript
{
  key: "original_relic_00001",
  label: "回血石",
  interaction: {
    possess: {
      triggers: [{
        when: "after",
        how: "make",
        key: "battleEnd",
        action: "healOwner"  // 新增：指定响应名称
      }]
    }
  },
  reaction: {  // 新增：响应定义
    healOwner: [{
      targetType: "owner",
      key: "heal",
      effect: [{ key: "heal", params: { value: 5 } }]
    }]
  }
}
```

### 对比当前结构

当前：
```typescript
triggers: [{
  when: "after",
  how: "make",
  key: "battleEnd",
  event: [{  // event 在 trigger 里
    targetType: "owner",
    key: "heal",
    effect: [...]
  }]
}]
```

新设计：
```typescript
triggers: [{
  when: "after",
  how: "make",
  key: "battleEnd",
  action: "healOwner"  // 只指定响应名
}]
reaction: {
  healOwner: [...]  // 响应内容在外面
}
```

## 核心优势

1. 所有权清晰
   遗物的行为在遗物自己身上，不在玩家的 trigger 里

2. 职责单一
   trigger 只负责"什么时候通知谁"
   reaction 负责"收到通知后做什么"

3. 复用性
   多个 trigger 可以共用一个 reaction

4. 调试友好
   listTriggers() 可以清晰显示"这个 trigger 来自哪个遗物，会触发哪个响应"

5. 保留数据驱动
   reaction 仍然是 event/effect 数据，不需要写函数
   mod 作者门槛不变

## 实施步骤

### 阶段1：抽取 event 执行逻辑

目标：把当前 `createTriggerByTriggerMap` 里的 event 执行逻辑抽成独立函数

文件：`src/core/objects/system/trigger/Trigger.ts`

新增函数：
```typescript
executeReactionEvents(params: {
  reactionEvents: TriggerEventConfig[]
  triggerEvent: ActionEvent
  trigger: RuntimeTrigger
  triggerEffect: Effect | null
}): void
```

这个函数负责：
  解析 targetType
  解析 mediumType
  解析 $triggerValue
  调用 doEvent

当前这段逻辑在 `createTriggerByTriggerMap` 的 callback 里（第135-186行）。

### 阶段4：扩展 RuntimeTrigger 结构

文件：`src/core/objects/system/trigger/Trigger.ts`

当前运行时触发器对象需要补充上下文：
```typescript
interface RuntimeTrigger {
  when: "before" | "after"
  how: "make" | "via" | "take"
  key: string
  owner: Entity      // 持有触发器的对象（玩家）
  source: Entity     // 提供触发器的对象（遗物/器官）
  action?: string    // 响应名称
  level?: number
  importantKey?: string
  onlyKey?: string
  callback: TriggerFunc
  remove: () => void
}
```

目的：reaction 执行器只需要 `triggerEvent + trigger` 就能拿到全部上下文。

### 阶段5：修改 createTriggerByTriggerMap

核心改动：

旧逻辑：
```typescript
callback(event, effect, triggerLevel) {
  // 直接遍历 item.event 并执行 doEvent
}
```

新逻辑：
```typescript
callback(event, effect, triggerLevel) {
  // 先检查 condition

  // 优先级1：如果 item.action 存在，则通知 source，执行 source.reaction[action]
  if (item.action && source.reaction?.[item.action]) {
    executeReactionEvents({
      reactionEvents: source.reaction[item.action],
      triggerEvent: event,
      trigger: runtimeTrigger,
      triggerEffect: effect,
    })
    return
  }

  // 优先级2：兼容旧写法，如果 item.event 存在，继续执行旧逻辑
  if (item.event) {
    executeReactionEvents({
      reactionEvents: item.event,
      triggerEvent: event,
      trigger: runtimeTrigger,
      triggerEffect: effect,
    })
  }
}
```

这样可以兼容新旧两套写法。

## 风险与注意事项

1. 兼容性
   必须保证旧写法继续工作，不能一次性全部迁移

2. 类型安全
   reaction 字段需要正确的类型定义，避免运行时错误

3. 调试工具
   listTriggers() 需要同步更新，显示 action 和 reaction 信息

4. 测试覆盖
   每个阶段都要测试，确保不破坏现有功能
