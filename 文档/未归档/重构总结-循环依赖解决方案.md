# 循环依赖解决方案总结

## 核心发现

**关键认知**：`import type` 不会造成运行时循环依赖，只有**值导入**（`import { ... }`）才会。

## 循环依赖链

找到的循环依赖路径：

```
Item.ts
  ↓ import { Entity } (值导入，继承需要)
Entity.ts
  ↓ import { Status } (值导入)
Status.ts
  ↓ import { Modifier } (值导入)
Modifier.ts
  ↓ (被 ItemModifier 继承)
ItemModifier.ts
  ↓ import { Item } (值导入) ← 循环！
Item.ts
```

## 解决方案

### 原则

1. **类型导入优先**：能用 `import type` 就用 `import type`
2. **接口打破循环**：只在值导入形成循环时使用接口
3. **继承必须值导入**：`extends` 关系无法避免值导入

### 具体修改

**ItemModifier.ts**：
```typescript
// 之前（值导入，形成循环）
import { Item } from "../../item/Item"

// 之后（类型导入，打破循环）
import type { Item } from "../../item/Item"
```

**原因**：ItemModifier 只需要 Item 的类型信息，不需要实例化 Item 类。

## 接口的用途

基于这次分析，接口应该用于：

### 1. 打破值导入循环依赖

当 A 需要值导入 B，但 B 又通过某条路径值导入 A 时：
- 定义 IB 接口
- A 改为值导入 IB 接口
- B 实现 IB 接口

### 2. 工厂模式（未来）

```typescript
// 工厂返回接口，隐藏具体实现
export function createEntity(map: EntityMap): IEntity {
    return new Entity(map)
}
```

### 3. Mod API（未来）

接口作为稳定的 API 契约，mod 开发者依赖接口而不是具体实现。

## 不需要接口的情况

### 1. 类型导入

```typescript
// 这样不会循环依赖，不需要接口
import type { Entity } from './Entity'
function foo(entity: Entity) { ... }
```

### 2. 单向依赖

```typescript
// Item 继承 Entity，这是单向的，不需要接口
export class Item extends Entity { ... }
```

### 3. 懒加载的效果函数

效果函数通过 effectMap 懒加载，不在模块初始化时导入，所以可以直接使用具体类。

## 经验教训

1. **先分析再动手**：应该先找出真正的循环依赖，而不是盲目修改
2. **理解工具**：`import type` 是解决大部分问题的关键
3. **接口不是万能的**：只在真正需要时使用，不要过度设计
4. **保留注释**：代码注释很重要，不要随意删除
