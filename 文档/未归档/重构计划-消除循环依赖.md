# 重构计划：消除循环依赖

## 问题概述

项目存在严重的循环引用问题，主要表现为：

1. 类直接互相引用（Entity ↔ ActionEvent ↔ Item）
2. 类型和值混用（同一个 import 既当类型又当值）
3. 缺少抽象层（具体类之间直接依赖）
4. 服务定位器模式缺失（到处直接 new）
5. 模块职责不清晰（typeGuards 依赖具体类，各种 Modifier 互相依赖）

## 目标架构

建立清晰的分层架构，依赖方向单向向下：

```
types (纯类型)
  ↓
interfaces (抽象接口)
  ↓
base (基础抽象类)
  ↓
implementations (具体实现)
  ↓
container (依赖注入)
```

## 重构阶段

### 阶段 0：准备工作（当前阶段）

目标：建立新的目录结构，不破坏现有代码

步骤：
  创建新的目录结构
    src/core/types/ - 纯类型定义
    src/core/interfaces/ - 抽象接口
    src/core/base/ - 基础抽象类
    src/core/implementations/ - 具体实现（暂时为空）

  创建 types/index.ts 作为类型中心
    提取所有纯类型定义
    EventParticipantType
    EntityType
    ItemType
    基础数据结构类型

  创建 interfaces/index.ts 作为接口中心
    IEntity - Entity 的接口
    IActionEvent - ActionEvent 的接口
    IItem - Item 的接口
    ITrigger - Trigger 的接口
    IStatus - Status 的接口

验证：
  新目录创建成功
  类型文件可以独立编译
  接口文件只依赖 types，不依赖具体实现

### 阶段 1：类型与值分离

目标：将所有 import 分为 import type 和 import value

步骤：
  审查所有文件的 import 语句

  将仅用作类型的 import 改为 import type
    例如：import type { Entity } from './Entity'

  将既用作类型又用作值的 import 拆分
    类型部分：import type { EntityType } from './types'
    值部分：import { createEntity } from './factory'

  使用 TypeScript 的 importsNotUsedAsValues 检查

验证：
  所有文件编译通过
  运行时行为不变
  使用 tsc --noEmit 检查类型错误

### 阶段 2：提取接口层

目标：为核心类创建接口，解耦具体实现

步骤：
  为 Entity 创建 IEntity 接口
    包含所有公共方法和属性
    不包含实现细节

  为 ActionEvent 创建 IActionEvent 接口

  为 Item 创建 IItem 接口

  为 Modifier 系列创建 IModifier 接口

  修改类型守卫函数
    typeGuards.ts 只依赖接口，不依赖具体类
    使用 participantType 等标识符判断类型

验证：
  接口文件只依赖 types 层
  类型守卫函数可以独立编译
  现有代码仍然正常工作

### 阶段 3：引入工厂模式

目标：消除直接 new 操作，使用工厂函数创建对象

步骤：
  创建 factories/ 目录

  创建 EntityFactory
    createEntity(map: EntityMap): IEntity
    内部处理具体类的实例化

  创建 ActionEventFactory
    createActionEvent(params): IActionEvent

  创建 ItemFactory
    createItem(map: ItemMap): IItem
    createCard(map: CardMap): ICard
    createRelic(map: RelicMap): IRelic

  逐步替换代码中的 new Entity() 为 createEntity()

验证：
  工厂函数可以正确创建对象
  对象行为与之前一致
  依赖关系更清晰

### 阶段 4：重构 Modifier 系统

目标：消除 Modifier 之间的循环依赖

步骤：
  创建 IModifier 基础接口
    apply(): void
    remove(): void
    getSource(): IEntity

  创建 ModifierManager
    统一管理所有修饰器
    提供注册、应用、移除接口

  重构各个具体 Modifier
    ItemModifier 只依赖 IItem 接口
    StatusModifier 只依赖 IStatus 接口
    不再互相导入

  使用组合而非继承
    Modifier 不再继承自 Entity
    使用组合关系持有 Entity 引用

验证：
  Modifier 系统功能正常
  器官获得/失去时修饰器正确应用/移除
  没有循环依赖

### 阶段 5：重构 Effect 系统

目标：解耦 Effect 和 Entity

步骤：
  创建 IEffect 接口

  创建 EffectContext
    包含 source, medium, target
    提供统一的上下文访问

  重构 EffectFunc
    只依赖 IEffect 和 EffectContext
    不直接导入具体的 Entity 类

  使用依赖注入传递依赖
    EffectFunc 通过参数接收所需服务
    不在函数内部直接导入

验证：
  所有效果函数正常工作
  伤害、治疗、抽卡等核心功能正常
  Effect 系统没有循环依赖

### 阶段 6：重构 Trigger 系统

目标：解耦 Trigger 和 Entity

步骤：
  创建 ITrigger 接口

  创建 TriggerContext
    包含 event, effect, triggerLevel
    提供统一的触发上下文

  重构 Trigger 类
    只依赖接口，不依赖具体类
    使用 TriggerContext 传递信息

  重构 TriggerMap 处理
    使用工厂函数创建触发器
    不在 Trigger 内部直接实例化 Entity

验证：
  触发器系统正常工作
  before/after, make/via/take 机制正常
  触发器级联正常

### 阶段 7：建立依赖注入容器

目标：统一管理对象创建和依赖关系

步骤：
  创建 Container 类
    register(key, factory)
    resolve(key)
    singleton(key, factory)

  注册核心服务
    EntityFactory
    ActionEventFactory
    ItemFactory
    ModifierManager
    EffectManager

  在应用启动时初始化容器
    main.ts 中配置容器
    preload 时注册所有服务

  逐步迁移代码使用容器
    从容器获取工厂
    从容器获取管理器

验证：
  容器可以正确解析依赖
  单例服务只创建一次
  应用启动正常

### 阶段 8：清理旧代码

目标：移除临时兼容代码，完成迁移

步骤：
  移除所有 console.log 调试代码

  移除旧的直接 import 语句

  统一使用新的架构
    所有创建通过工厂
    所有依赖通过接口
    所有服务通过容器

  更新文档
    更新 CLAUDE.md
    更新架构文档
    添加迁移指南

验证：
  所有功能正常
  没有循环依赖
  代码质量提升
  TypeScript 编译无错误

## 关键原则

代码质量第一
  即使为了快速实现，也不牺牲代码质量
  重构优于重复，抽象优于复制粘贴

一次一个问题
  每个阶段专注解决一个问题
  等待验证通过后再进入下一阶段

用户引导方向
  每个阶段完成后等待用户反馈
  根据用户指示调整计划

渐进式重构
  不破坏现有功能
  新旧代码可以共存
  逐步迁移，降低风险

接口隔离
  依赖接口而非实现
  使用 import type 而非 import
  通过工厂创建对象

单向依赖
  types ← interfaces ← base ← implementations
  下层不能依赖上层
  同层之间通过接口通信
